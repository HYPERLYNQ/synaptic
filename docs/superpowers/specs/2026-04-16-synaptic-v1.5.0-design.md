# Synaptic v1.5.0 ŌĆö Automatic Capture & Smart Recall

**Date:** 2026-04-16
**Status:** Approved
**Builds on:** v1.4.0 (`7afb254` ŌĆö auto-save hooks + `/checkpoint` slash command)
**Motivating incident:** RTX 5090 tracker session 2026-04-16 03:17 produced a high-value narrative handoff. By 2026-04-17 20:32, two content-less aggregation handoffs from the Stop hook had displaced it in SessionStart auto-recall. User returned 2026-04-16, auto-recall surfaced `"hyperlynq (27 entries ŌĆö insight:21, progress:4, decision:2)"` instead of the useful handoff. Phases 1ŌĆō4 of the tracker were shipped in prior sessions but no checkpoint was fired for any of them ŌĆö auto-save is a directive, not a hook.

## Problem Statement

Synaptic v1.4.0 has four compounding reliability issues:

1. **Stop hook produces content-less handoff entries.** Every session end writes a bare aggregation (`N entries ŌĆö insight:X, progress:Y`) as a `handoff` row. These have no narrative value but are picked up by recall.
2. **SessionStart recall is blindly recency-ordered.** The most recent handoff wins regardless of content quality. Empty aggregations from (1) crowd out genuine narrative handoffs.
3. **Auto-save isn't a hook, it's a directive.** `context_save` is supposed to fire "immediately" when the assistant notices significant moments, but in practice significant moments (phase commits, plan writes, save-intent utterances) slip past unless the assistant proactively remembers. No deterministic capture layer exists.
4. **Recall has no project-awareness.** `cwd` isn't used to filter or rank entries. A handoff from a different project can outrank a relevant one from the current project.

## Scope

Seven deliverables, ordered by dependency:

1. New `checkpoint` entry type + schema migration
2. Stop hook: gated narrative synthesis (no more empty counts)
3. New PostToolUse hook: auto-checkpoint on commits and plan/spec writes
4. New UserPromptSubmit hook: regex-based save-intent detection
5. SessionStart hook: smart scoring + project-awareness for panel selection
6. New slash commands: `/load <name>` and `/checkpoints`
7. New MCP tool: `context_load`

## Section 1: `checkpoint` Entry Type

**Schema migration** (`scripts/migrate-v1.5.0.cjs`):

```sql
ALTER TABLE entries ADD COLUMN name              TEXT;
ALTER TABLE entries ADD COLUMN summary           TEXT;
ALTER TABLE entries ADD COLUMN pinned            INTEGER DEFAULT 0;
ALTER TABLE entries ADD COLUMN projectRoot       TEXT;
ALTER TABLE entries ADD COLUMN referencedEntryIds TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS idx_entries_name ON entries(name) WHERE name IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_entries_projectRoot ON entries(projectRoot);
CREATE INDEX IF NOT EXISTS idx_entries_pinned ON entries(pinned) WHERE pinned = 1;
```

**Semantics:**

- `name` (TEXT UNIQUE, nullable) ŌĆö slug identifier for checkpoints, e.g. `rtx-tracker-phase-5-start`. NULL for non-checkpoint types. Unique index applies only when non-NULL.
- `summary` (TEXT, nullable) ŌĆö one-line Ōēż120-char hook shown in SessionStart panel.
- `pinned` (INTEGER, default 0) ŌĆö 1 means the entry survives aggressive archiving and ranks higher in recall. Default 1 for checkpoints, 0 for other types.
- `projectRoot` (TEXT, nullable) ŌĆö absolute path, git toplevel at save time if available, else `cwd`.
- `referencedEntryIds` (TEXT, JSON array, nullable) ŌĆö ids of related entries saved in the same session.

`content` stays the narrative body (same as existing types). `type` adds `'checkpoint'` to the existing enum.

**Differentiation from existing types:**
- `progress` ŌĆö factual status snapshot. Not pinnable, not named, not loadable.
- `handoff` ŌĆö session-end narrative. Not pinnable, not named, not loadable. Written by Stop hook.
- `checkpoint` ŌĆö named, pinnable, loadable save point. Written by `/checkpoint`, PostToolUse, or UserPromptSubmit.

**Migration backfill:**
- For each existing entry tagged with a known project (e.g. `rtx-5090-tracker`, `synaptic`), set `projectRoot` to the inferred absolute path.
- For existing `handoff` entries where `length(content) < 100`: archive (these are legacy empty-count rows).
- For v1.4.0 checkpoints saved via the slash command (if stored as `insight` with a `checkpoint` tag), convert `type` to `checkpoint` and derive `name` from an existing `checkpoint-<slug>` tag if present.

## Section 2: Stop Hook ŌĆö Gated Narrative Synthesis

**File:** `src/hooks/stop.ts` (modify existing)

Current behavior: every session end writes a handoff with body `"**<project>** (N entries ŌĆö insight:X, progress:Y, decision:Z)"`. No narrative.

New behavior:

```
on Stop:
  events = countMeaningfulSessionEvents(sessionId)
  if events.total < 1:
    log "session had no meaningful events, skipping handoff"
    return
  narrative = synthesizeNarrative(sessionId)  // existing transcript-scan logic
  if narrative.length < 100:
    log "synthesis produced thin narrative, skipping handoff"
    return
  saveEntry({
    type: 'handoff',
    content: narrative,
    projectRoot: detectProjectRoot(cwd),
    tags: [...existingTags],
    tier: 'working',
  })
```

`countMeaningfulSessionEvents(sessionId)` counts:
- `git_commit` entries saved during session
- `checkpoint` entries saved during session
- `decision` entries saved during session
- Any `Write` tool calls to `docs/superpowers/plans/` or `docs/superpowers/specs/`

Threshold: `events.total >= 1`. Below threshold ŌåÆ write nothing.

The existing content-less `(N entries ŌĆö insight:X ŌĆ”)` aggregation body is removed entirely. It was never useful; it was a placeholder that shipped.

## Section 3: PostToolUse Hook ŌĆö Auto-Checkpoint

**New file:** `src/hooks/post-tool-use.ts`

**Hook config addition** (installed by the plugin, mirrors v1.4.0 installer pattern):

```json
{
  "PostToolUse": [{
    "matcher": "Bash|Write|Edit",
    "hooks": [{
      "type": "command",
      "command": "node --no-warnings /home/hyperlynq/synaptic/build/src/hooks/post-tool-use.js",
      "timeout": 3
    }]
  }]
}
```

**Trigger patterns and content derivation:**

| Pattern | Match condition | Checkpoint name | Summary | Content |
|---|---|---|---|---|
| Commit | `tool=Bash` AND command matches `^\s*git commit\b` AND NOT `--amend` AND NOT `-i` | slug of commit subject (first 60 chars) | first line of commit msg | `<commit_msg>\n\n${git diff --stat HEAD~1..HEAD}` |
| Plan write | `tool in [Write, Edit]` AND path matches `docs/superpowers/plans/.+\.md$` | slug of basename, date prefix stripped | first `#` heading of file (or basename if no heading) | `Plan landed at <path>.\n\n<first 200 chars of body>` |
| Spec write | `tool in [Write, Edit]` AND path matches `docs/superpowers/specs/.+\.md$` | slug of basename, date prefix stripped | first `#` heading of file | `Spec landed at <path>.\n\n<first 200 chars of body>` |

**Dedupe rules** (applied before save):

1. If a `checkpoint` entry exists with the same `commitSha` (stored as tag `sha:<sha>`) or same `filePath` (stored as tag `path:<abs-path>`) and `sessionId` == current session ŌåÆ skip.
2. If a manual `checkpoint` was saved via `/checkpoint` or UserPromptSubmit within the last 60 seconds ŌåÆ skip (user intent supersedes auto).
3. Commit must touch at least one file outside `.synaptic/` ŌĆö skip auto-checkpoints that only modify Synaptic's own context db.

**Performance:**

- Hook must complete in <500ms p99. It runs after every tool call.
- No embedding in the hot path. Embeddings for new checkpoints are backfilled by SessionStart (existing pattern).
- Classification (slug derivation, summary extraction) is synchronous regex work on strings already in memory; no IO beyond the SQLite INSERT.
- Hook timeout is 3s (safety margin, not target).

**Shelling out to git:** use `execFile` (via the existing `execFileNoThrow` helper if available), never shell `exec`. Commit sha derivation: `git rev-parse HEAD` with no user-supplied arguments. Diff-stat derivation: `git diff --stat HEAD~1..HEAD` with no user-supplied arguments.

## Section 4: UserPromptSubmit Hook ŌĆö Save-Intent Detection

**New file:** `src/hooks/user-prompt-submit.ts`

**Hook config addition:**

```json
{
  "UserPromptSubmit": [{
    "hooks": [{
      "type": "command",
      "command": "node --no-warnings /home/hyperlynq/synaptic/build/src/hooks/user-prompt-submit.js",
      "timeout": 2
    }]
  }]
}
```

**Detection:** single compiled regex, case-insensitive:

```
\b(save (progress|this|state|checkpoint|it|that)|wrap (up|it up)|/checkpoint\b|let'?s checkpoint|checkpoint (this|that|it)|remember this|note this|bookmark this|save where we are|mark this)\b
```

**On match:**
- Emit a checkpoint with:
  - `name` = `user-intent-<timestamp>` (deterministic slug; will be superseded by any follow-up explicit `/checkpoint <name>`)
  - `summary` = "User-requested checkpoint"
  - `content` = a synthesized narrative from the session so far (reuses `synthesizeNarrative`)
  - `referencedEntryIds` = all entries saved in this session
- Return the hook response schema Claude Code expects (continue=true, no context injection).

**On no match:** exit with code 0, no output. Do not block the prompt.

**Performance:**
- Target <50ms p99. Hook fires on every user turn; UX cost of regex miss is zero latency.
- Regex-only, no embedding. False negatives recoverable via explicit `/checkpoint`.
- False positives are cheap: duplicate checkpoint, which the PostToolUse dedupe rule will catch on the next commit anyway.

**Why not embedding classification:**
- Loading an embedding model adds 50ŌĆō200ms per invocation on every user turn.
- v1.4.0 already bundles ONNX for `context_search`; we could reuse the warmed model in-process ŌĆö but UserPromptSubmit runs as a standalone Node child process per Claude Code hook architecture. No warm cache across invocations.
- Regex with a curated phrase list captures >95% of save-intent utterances. The long tail (ambiguous meta-phrases) is better served by explicit slash commands.

## Section 5: SessionStart Hook ŌĆö Smart Scoring

**File:** `src/hooks/session-start.ts` (modify existing)

**Current behavior:** picks the most recent handoff. No scoring, no filtering.

**New behavior:**

```
on SessionStart:
  currentProjectRoot = detectProjectRoot(cwd)
  candidates = query({
    type: ['handoff', 'checkpoint'],
    archived: false,
    content_length_min: 100,  // filter legacy empties
    limit: 30,
  })
  ranked = rankEntries(candidates, currentProjectRoot)
  top3 = ranked.slice(0, 3)
  emit panel with top3
```

**`rankEntries` scoring function** (`src/lib/scoring.ts` ŌĆö pure, unit-testable):

```
score(entry) =
  0.35 * min(entry.content.length / 500, 1.0)
+ 0.35 * projectMatch(entry, currentProjectRoot)
+ 0.15 * (entry.pinned ? 1 : 0)
+ 0.15 * exp(-hoursOld(entry) / 72)

projectMatch(entry, current):
  if entry.projectRoot == current:                    return 1.0
  if entry.tags overlap knownProjectTags(current):    return 0.3
  else:                                               return 0
```

Rationale for weights:
- **Length (0.35)** and **project match (0.35)** are co-equal ŌĆö content quality and relevance to current work matter most.
- **Pinned (0.15)** ŌĆö a moderate boost so user-pinned items don't lose to marginally newer unpinned ones, but fresher project-matched content still wins when it's clearly better.
- **Recency (0.15)** ŌĆö a tiebreaker, not the primary signal. 3-day half-life via `exp(-hours/72)`.

**Content-length filter:** entries with `length(content) < 100` are excluded *before* scoring. This is the safety net for any v1.4.0 empty-count handoffs the migration didn't catch.

## Section 6: `/load` and `/checkpoints` Slash Commands

**New file:** `commands/load.md`

```markdown
---
description: Load a saved checkpoint into current context
---

Call the `context_load` MCP tool with `name={{args}}`. Inject the returned narrative and references into the current conversation as context.
```

**New file:** `commands/checkpoints.md`

```markdown
---
description: List recent checkpoints (project-scoped)
---

Call the `context_list` MCP tool filtered to `type=checkpoint`, sorted by recency, scoped to the current project root. Show top 10 with name, summary, and relative timestamp.
```

**New MCP tool:** `context_load` (in `src/mcp/tools.ts`):

```
input: { name: string }
output: {
  checkpoint: { name, summary, content, projectRoot, createdAt },
  references: [{ id, type, date, contentPreview }]  // first 120 chars each
}
```

Lookup order:
1. Exact `name` match on non-archived checkpoints
2. Fuzzy slug match (Levenshtein distance Ōēż 2 on normalized form)
3. Fuzzy content search (existing `hybridSearch`) among checkpoints only

If multiple candidates tie: return list of matches with ask-to-disambiguate flag; Claude formats this as "Multiple matches found ŌĆö please run `/load <exact-name>`."

**Injection format** (Claude's job, not the MCP tool's):

```
## Loaded checkpoint: <name>
_Saved <relative-time> from <projectRoot>_

<content>

### Referenced entries
- [<id>] (<type>, <date>): <contentPreview>
...
```

**Token cap:** total injected content Ōēż ~4000 tokens. Truncate `references` first; if narrative itself exceeds the cap, truncate it with a note.

## Section 7: Project-Awareness

**New file:** `src/lib/project-root.ts`

Responsibilities:
- `detectProjectRoot(cwd)` ŌĆö spawn `git rev-parse --show-toplevel` via `execFile` (never shell-exec), `cwd: cwd`, stdin ignored, stderr suppressed. If the process exits non-zero or the output is empty, return `cwd` unchanged. No user-controlled arguments are ever passed to git.
- `knownProjectTags(projectRoot)` ŌĆö derive candidate tags from the directory basename: the basename itself, the basename with dashes stripped, and each dash-separated segment of length Ōēź 3. Used for secondary tag-overlap matching when an entry predates the `projectRoot` field.

Every new entry (any type) records `projectRoot` at save time. The migration backfills best-effort values on existing entries.

## Migration (`scripts/migrate-v1.5.0.cjs`)

Runs once on package upgrade (wired into `postinstall` per Pattern D).

Steps:
1. Check schema version. Exit if already at v1.5.0.
2. `ALTER TABLE entries` ŌĆö add the five new columns + indexes.
3. **Backfill `projectRoot`** on existing entries via tag heuristics:
   - `rtx-5090-tracker` tag ŌåÆ `/home/hyperlynq/rtx-5090-tracker`
   - `synaptic` tag ŌåÆ `/home/hyperlynq/synaptic`
   - No match ŌåÆ NULL (will be populated on next save)
4. **Archive legacy empty handoffs:**
   ```sql
   UPDATE entries SET archived = 1 WHERE type = 'handoff' AND length(content) < 100;
   ```
5. **Convert v1.4.0 slash-command checkpoints** (if any) from `type='insight'` with `checkpoint` tag to `type='checkpoint'`, deriving `name` from the existing tag pattern.
6. **Rebuild FTS5 index** (existing pattern).
7. Update schema version row to `1.5.0`.
8. Print report: counts of migrated rows per step.

Dry-run flag: `--dry-run` prints what would change without writing.

## Out of Scope (Deferred to v1.6.0)

- **Embedding-based intent classifier** for UserPromptSubmit ŌĆö regex is the right call for v1.5.0. Revisit if we see meaningful false-negatives in production.
- **Named checkpoint chains** (`chain:<id>` linking multiple checkpoints into an arc). The `referencedEntryIds` field already enables bundle-loading; arcing across bundles is a v1.6 feature.
- **Checkpoint diffing** (`/compare <name1> <name2>`). Useful but not load-bearing for the core "never forget to save" goal.
- **Non-git project roots via `.synaptic-project` sentinel file.** v1.5.0 uses git rev-parse with cwd fallback ŌĆö covers ~all real cases. Sentinel file is a nice-to-have for worktree edge cases.

## Success Criteria

1. A user who commits code and restarts Claude Code sees their work summarized in the SessionStart panel ŌĆö with zero manual saves.
2. The RTX 5090 scenario: if tonight we commit phase 5, tomorrow's SessionStart panel shows "phase-5-complete" as the top checkpoint, not a count aggregation.

5. A user saying "save progress" mid-session results in a checkpoint without them typing a slash command.
